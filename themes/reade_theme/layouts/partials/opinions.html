Opinions

Client

For large commercial applications I recommend using React as a front-end. It has a large community, as well as a huge ecosystem of libraries and solutions.

TypeScript is also a must. I can't think of a single good reason not to use it, even with existing JavaScript projects.

For smaller teams I recommend using Elm. It allows you to build the same thing in half the time. Plus you get Algebraic Data Types, type safety and state management baked in for FREE.

State management

Apollo is my go-to solution. GraphQL truly is a no-brainer.

Redux is philosophically sound, however the excessive boilerplate is a huge productivity killer. In addition, Apollo offers the exact same uni-directional data flow as Redux, for a fraction of the code.

Prisma is also amazing as a GraphQL backend service, especially if you have some of that sweet VC money to spend. Otherwise graphql-yoga is my go-to GraphQL server setup (built upon Node.js).

Server

Node.js and Elixir satisfy everything you could possibly want from an application backend.

Meteor.js is great for building quick MVP projects, especially if you need real-time functionality. Phoenix Framework is also amazing for achieving the exact same thing. Both have their strengths and weaknesses.

Express.js is average at best. Feathers.js is a great alternative (that's built upon Express.js) and that provides real-time functionality. Nest.js is also great for building scalable APIs.

There is also the fabulous Parse Server which I absolutely LOVE. It comes baked with so much functionality, including real-time data, user authentication, push notifications and so. much. more.

Databases

PostgreSQL if you want to do it right.

MongoDB if you're walking down the street and a stranger happens to pull a gun on you, as you find yourself compelled to make a seemingly irrational decision in the ensuing panic, startling you into an uncontrollable mania of fear and destitution.

...and for everything else, there's Mastercard.

DevOps

To be quite frank, AWS is overkill in 95% of cases. In most cases, you can avoid having to rely on AWS if you use the right tech to begin with.

A simple VPS via Digital Ocean is going to be more than enough for a majority of clients, and you'll save a bucketload of time and $$$ in the process. Ubuntu, Zsh and Nginx are your friends.

Of course, I totally understand the business rationale for using AWS for large companies where things like permissions and scalability are important, but I guess I'm just a bit of a startup guy.

Github for version control, Docker for application container management, Buildkite for continuous CI and Terraform for server configuration/setup.

Content

For general blogs and basic websites I use Hugo (which is what this website is built upon) along with Netlify for the most delightful deployment process of your lifetime.

If you require CMS functionality, then tacking on Forestry.io may serve you well. Oh, and I absolutely LOATH Wordpress. You'd derive more enjoyment chewing glass and spitting it out again.

Mobile

For cross-platform mobile applications I recommend using Flutter.

Although for long-term projects projects I would consider writing them in their respective iOS and Android native languages, as you'll have a better product, as well as save more time overall.

React Native isn't necessarily the worst experience in the world, however it's a fraction as nice as Flutter. Even if you're already fluent with React, I'd bet you'd still come out ahead learning Flutter, especially if you've never built a mobile application before.

One Signal is great for push/mobile notifications, Fastlane for Continuous Integration and Bitrise for Continuous Deployment.
