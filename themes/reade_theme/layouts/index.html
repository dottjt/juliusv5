Client Frameworks

For large commercial applications I recommend using React as a front-end, along with TypeScript.

TypeScript is a must with React. I can't think of a single good reason not to use it, even with existing projects.

For small teams I recommend using Elm. It allows you to build the same thing in half the time. You get type safety + state management baked in for free and your app will run much faster.

I'd even recommend Elm for large commercial projects, but you'd never convince management or even senior staff entrenched in their old ways.

State management

I much prefer using Apollo over Redux.

Redux boilerplate is a huge productivity killer, and Apollo provides the same clean uni-directional data flow at a fraction of the code.

Server Frameworks

Node.js and Elixir satisfy everything you could possibly need for a backend.

Meteor.js is great for building quick MVP projects, especially if you need real-time functionality.

Phoenix Framework is also amazing for achieving the same thing. Difference being that your project will actually scale and you won't have to rebuild it again within a few years (at the cost of taking a little bit more time to develop).

Express.js is average at best. Feathers.js is a good alternative (it's built upon Express.js) that provides real-time functionality + some great services. Nest.js is great for building scalable APIs.

There is also the incredible Parse Server self-hosted BAAS (built on node.js) which is what I primarily use to build my backends. It comes baked with so much functionality, it really is a huge time saver.

GraphQL

In terms of GraphQL servers, Prisma is amazing BAAS if you have the money. graphql-yoga is generally my go-to.

Web Infrastructure

To be quite frank, AWS is almost always overkill.

In most cases, you can avoid having to use AWS if you use the right tech to begin with (Elixir for concurrent connections etc.)

In most cases, a simple VPS via Dropbox is going to be more than enough, and you'll save a bucketload of money.
